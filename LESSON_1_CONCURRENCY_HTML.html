Why should I care

Concurrency as a practical problem rather than an abstract one. Below are real life scenarios where not using concurrency correctly with have catastrophic results

You already know data structures. Concurrency raises the next question: what happens when more than one actor touches the same structure at the same time?

The Runnable interface

Before looking at threads themselves it helps to understand what a thread actually runs. That unit of work is described by java.lang.Runnable — one of the oldest and most important interfaces in Java.

🔷

What Runnable is

Runnable is a functional interface — it has exactly one abstract method:

@FunctionalInterface
public interface Runnable {
    void run();
}

The @FunctionalInterface annotation is a compile-time guard: if someone accidentally adds a second abstract method the compiler refuses to build. It also signals to readers that this interface is safe to use with lambda expressions and method references.

One rule: run() takes no arguments and returns nothing. All inputs must come in through the enclosing scope (captured variables must be effectively final).
✏️

Four equivalent ways to write a Runnable

Runnable r = new Runnable() {
    @Override
    public void run() {
        System.out.println("hello");
    }
};
// Lambda — most common modern style
Runnable r = () -> System.out.println("hello");
// Method reference — works when a static or
// instance method matches the () -> void shape
Runnable r = MyClass::doWork;
class Greeter implements Runnable {
    @Override
    public void run() {
        System.out.println("hello");
    }
}

Runnable r = new Greeter();

Why "functional interface" matters

Capability What it means in practice
Lambda expression Any () -> void lambda can be assigned to a Runnable variable without boilerplate.
Method reference Any zero-argument void method can be used directly: Thread t = new Thread(obj::method).
Works everywhere threads are accepted new Thread(r), executor.submit(r), CompletableFuture.runAsync(r) — they all accept Runnable.
Composable You can store, pass, and wrap Runnable objects just like any other value — useful for task queues and decorators.
Captured variables must be effectively final. A lambda captures the reference, not a copy of a mutable variable. Mutating a captured variable from inside run() while another thread reads it is a data race — exactly the problem the rest of this lesson addresses.

What is a thread?

A process is a running program. A thread is one path of execution inside that process. A Java program can have many threads working at once.

🧵

Simple mental model

Imagine a restaurant kitchen. One cook can only do one thing at a time. Several cooks can work in parallel, but they still share the same kitchen, same ingredients, and same order board.

The Thread class

Java's built-in java.lang.Thread class is the foundation of every thread. It bundles the lifecycle management, scheduling hints, and the unit of work (a Runnable) into one object.

🏗️

Two ways to create a thread

class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Hello from " + getName());
    }
}

MyThread t = new MyThread();
t.setName("worker-1");
t.start();          // schedules the thread
Runnable task = () ->
    System.out.println("Hello from "
        + Thread.currentThread().getName());

Thread t = new Thread(task, "worker-1");
t.start();          // schedules the thread
Prefer the Runnable approach. It keeps the task logic decoupled from thread mechanics and is compatible with ExecutorService.
🔄

Thread lifecycle

A thread moves through states managed by the JVM and the OS scheduler.

NEW Created but start() not yet called.
RUNNABLE start() called; running or waiting for CPU time.
BLOCKED Waiting to acquire a synchronized lock.
WAITING Waiting indefinitely via wait() or join().
TIMED_WAITING Sleeping via sleep(ms) or join(ms).
TERMINATED run() has returned or an exception escaped.

Key methods

Method What it does Notes
start() Registers the thread with the JVM scheduler and eventually calls run(). Never call run() directly — that executes on the current thread, not a new one.
run() Contains the code to execute in the new thread. Override it or supply a Runnable. Called by the JVM after start(); do not call it yourself.
sleep(long ms) Pauses the current thread for at least the given milliseconds. Static method. Does not release any held locks. Throws InterruptedException.
join() Blocks the calling thread until this thread finishes. Useful for waiting on results. Optional timeout overload: join(long ms).
interrupt() Sets the interrupt flag on the thread. Causes a sleeping/waiting thread to throw InterruptedException; a running thread must check isInterrupted().
isInterrupted() Returns true if the thread's interrupt flag is set. Does not clear the flag. Use inside long-running loops to respond to cancellation.
isAlive() Returns true if the thread has been started and has not yet terminated. Useful for quick liveness checks before calling join().
Thread.currentThread() Static factory that returns a reference to the thread currently executing the call. Works from any context — handy for logging the thread name inside a Runnable.
getState() Returns the current Thread.State enum value. Values: NEW, RUNNABLE, BLOCKED, WAITING, TIMED_WAITING, TERMINATED.

Key fields & properties

Property Getter / Setter Purpose
Name getName() / setName(String) Human-readable label for the thread. Invaluable in logs and stack traces.
Priority getPriority() / setPriority(int) Hint to the OS scheduler. Range: MIN_PRIORITY (1) → NORM_PRIORITY (5) → MAX_PRIORITY (10). Not a guarantee.
Daemon flag isDaemon() / setDaemon(boolean) Daemon threads are silently killed when all non-daemon threads finish. Must be set before start().
ID getId() A unique positive long assigned by the JVM. Read-only; useful for logging.
💡

Putting it together — sleep, join, and naming

Thread worker = new Thread(() -> {
    System.out.println(Thread.currentThread().getName() + " starting");
    try {
        Thread.sleep(500);           // pause for 500 ms
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt(); // restore flag
    }
    System.out.println(Thread.currentThread().getName() + " done");
}, "background-worker");

worker.setDaemon(false);   // keep JVM alive until this thread finishes
worker.setPriority(Thread.NORM_PRIORITY);
worker.start();

System.out.println("Main thread continuing...");
worker.join();             // wait for worker to finish
System.out.println("All done");
Always restore the interrupt flag after catching InterruptedException so that callers higher up the stack can still detect the cancellation.

Shared mutable state

If multiple threads can access a value and at least one of them can change it, that value is shared mutable state.

Shared thing Why it is risky Common symptom
int counter Increment is not one indivisible step. Lost updates
ArrayList<Task> Concurrent structural changes are unsafe. Wrong size, inconsistent contents, occasional exceptions
HashMap<K,V> Simultaneous reads and writes may produce inconsistent behavior. Missing or stale values
PriorityQueue<Task> Ordering logic is fine, but the queue itself is not thread-safe. Broken scheduling under concurrent access
Important: “It worked when I ran it” is not evidence that the program is correct. Race-condition bugs often hide until timing changes.

Why counter++ is dangerous

This is the central idea to slow down on. People usually assume counter++ is one operation. It is not.

The hidden steps

get current value
→
Compute
add 1
→

store new value
Atomic means the operation behaves like one indivisible step. Plain count++ is not atomic.
// Thread A reads 7
// Thread B reads 7
// Thread A writes 8
// Thread B writes 8

// Final value = 8
// Expected value = 9

Both threads did real work, but one update disappeared because both started from the same old value.

Fixing the problem with synchronized

The simplest correct fix at this stage is to protect the critical section so that only one thread enters it at a time.

public synchronized void increment() {
    count++;
}
Meaning: on the same object, only one thread at a time can execute this synchronized method.

Critical section

A critical section is the part of code that touches shared mutable state and must not be interleaved in the wrong way with another thread doing the same thing.

  • Keep it as small as possible.
  • Protect the read-modify-write sequence together.
  • Do not lock more code than you need.

Connecting concurrency back to collections

This course is still about data structures. Multithreading changes not only how we write code, but which structure we should trust in a concurrent setting.

Unsafe collection example

List<Task> tasks = new ArrayList<>();

tasks.add(new Task("T-1", Priority.HIGH));
// another thread also mutates the same list

An ArrayList is great for many single-threaded use cases. It is not built for many threads mutating it at once.

Safer intro-level option

List<Task> tasks = Collections.synchronizedList(
    new ArrayList<>()
);

This is not the end of the story, but it is a good first step for lesson 1. Later lessons will introduce concurrent collections and blocking queues.

Need Structure Concurrency note
Look up tasks by id Map Later: ConcurrentHashMap
Process highest priority first PriorityQueue Not thread-safe; later: PriorityBlockingQueue
Pass work in first-in, first-out order Queue Later: BlockingQueue
Simple sequential list ArrayList Fine for one thread, risky for shared mutation

Let's talk

Common misconception: “But Java is fast, so it should handle this.”
Speed is not the issue. The issue is interleaving. The CPU may switch between threads in many possible orders, and some orders break correctness.
Why this matters for data structures, not just counters
A counter is the simplest example of a shared mutable value. Collections are just larger, richer shared mutable structures. If you understand the counter, you can extend the idea to lists, maps, and queues.
